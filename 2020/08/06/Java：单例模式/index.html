Java：单例模式

1. 什么是单例模式？

2. 那么为什么需要单例？

原因之一： 是为了节省内存、节省计算。

说明：很多情况下，我们只需要一个实例就够了。如果出现了更多的实例，反而是一种浪费。

举例：

​ 我们就拿一个初始化比较耗时的类来说，在这个类构造的时候，需要查询数据库，并对查到的数据做大量的计算。所以在第一次计算的时候，我们花了很多时间来初始化这个对象.

​ 但是，假设我们数据库里的数据是不变的，并且把这个对象保存在了内存中。那么以后就可以用同一个实例。这种场景下每次都新生成一个实例，很没有必要。

原因之二：保证结果的正确性

比如：

​ 我们需要一个全局的计数器用来统计人数，如果有多个实例反而会造成混乱。

​ 另外，就是为了方便管理，很多工具类只需要一个实例。这是我们只需要通过一个统一的入口，比如通过 getInstance 方法，就可以获取到这个单例，这是很方便的。

​ 过多的实例不但没有帮助，反而会让人眼花缭乱。

3. 单例模式的应用场景

3.1 无状态的工具类

​ 比如日志工具类、字符串工具类…等等。

日志工具类： 我们需要的只是让它帮我们记录日志信息，除此之外，并不需要在它的实例对象上存储任何状态。这时候我们就只需要一个实例对象就可以了。

3.2 全局信息类

​ 比如全局计数、环境变量…..等等。

全局计数：比如我们在一个类上记录网站的访问次数，而且不希望有的访问被记录在对象A上，而有的却被记录在对象B上，这时我们就可以让这个类成为单例。在需要的时候拿出来用就可以了。

​ 对于全局的环境变量类也是如此。

4. 常见的单例模式写法

共有以下五种写法：

饿汉式、懒汉式、双重检查式、静态内部类式、枚举式

以下根据写法的难以层次来逐层递进：

4.1 饿汉式

​ 用static 修饰我们的实例，并且把构造函数用 private 修饰，这种写法比较简单，在类装载的时候就实现了实例化，避免了线程同步的问题。

​ 缺点： 在类装载的时候就及逆行了实例化，没有达到懒加载的目的。

​ 问题：如果从始至终都没有使用过这个实例，就可能会造成内存的浪费。

单例模式变种： 静态代码块式

​ 同样的，静态内部类式也是在类装载的时候就执行了静态代码块中的代码。完成了实例的初始化。

​ 其优缺点与构造器式相同。

4.2 懒汉式

​ 这种方式只有在 getInstance() 方法被调用时才会去实例化我们的对象， 起到了懒加载的效果。

​ 但是，该方式只能在单线程的环境中使用，如果在多线程环境中使用，一个线程进入了 if(Singleton == null) 判断语句块还没来得及执行，另一个线程也通过了这个判断语句。这时，就会多次创建实例，所以这里需要注意：在多线程环境下，不能使用这种方式，这是错误的写法。

​ 问题：每个线程在向获得类的实例的时候，执行 getInstance 方法都要进行同步，多个线程不能同时访问，而这在大多数情况下时没有必要的。

优化：缩小同步范围

​ 为了提高效率，缩小同步的范围，可以将 synchronized 关键字从方法上移除了。然后再把关键字放到方法内部，采用代码块的形式来保护线程安全。但，这种方式也存在问题：

​ 问题： 假如一个线程进入了第一个 if (singleton == null) 判断语句块还没来的及往下执行，而另一个线程也通过了这个判断语句，此时就会产生多个实例。

​ 解决方案： 双重检查模式

4.3 双重检查式

​ 在双重检查式代码中，我们进行了两次 if (singleton == null) 的判断，就可以保证线程安全了，这样实例代码化代码只用调用一次，后面再次访问的时候，只会判断第一次的 if (singleton == null) 就可以了。然后会跳过整个 if 块儿 ， 直接 return 实例化对象。

​ 优点： 这种写法，不仅线程安全，而且延迟加载，效率也更高。

关于 double-check ? 去掉第二次的 check 行不行?

​ 这时我们要考虑这样一种情况，有两个线程同时调用 getInstance 方法，并且由于 singleton 是空的，所以两个线程都可以通过第一层的 if 判断，然后由于锁机制的存在，会有一个线程线进入同步语句，并进入第二层 的 if 判断 ；而另一个线程就会在外面等待。

​ 不过当第一个线程执行完 new LazySingleton() 语句后就会退出 synchronized 的保护区域。这时，如果没有第二重 singleton == null 的话，那么第二个线程也会创建一个实例，这就破环了单例。这是肯定不行的。

​ 而对于第一个 check 而言，如果去掉它，那么所有的线程都会串行执行，效率低下。

关于 volatile 关键字的使用？

​ 添加该关键字修饰的原因，主要就在于 singleton = new LazySingleton(); 这句话，它并不是原子操作，事实上，在 JVM 中这句话至少做了 3 件事；

  1. 给 singleton 分配内存空间
  2. 调用 LazySingleton 的构造函数等来初始化 singleton
  3. 将 singleton 对象指向分配的内存空间（执行完这步 singleton 就不是 null 了）

​ 以上的 3 步的执行顺序在 JVM 中是不能保证被正确执行的，也就是说最终的执行顺序可能是 ：1-2-3 ，也可能是 1-3-2 ，如果是 1-3-2 的话，那么在执行完第3步后 ,singleton 就不是 null 了，而此时第 2 步并没有被执行。

​ 假设此时线程 2 进入了 getInstance 方法，由于此时的 singleton 不是 null 了，所以它就会通过第一重的检查并且直接返回，而实际上这个时候的 singleton 并没有完全地完成初始化。所以使用这个实例的时候就会报错。

4.4 静态内部类式

​ 它与饿汉式的方法类似，都采用了类装载的机制，来保证我们初始化实例时只有一个线程，所以 在这里 JVM 帮助我们保证了线程的安全性。

与饿汉式的区别：

​ 饿汉式的特点是，只要 Singleton 这个类被加载了，就会实例化单例对象，而静态内部类方式在 Singleton 类被装载时，并不会立刻实例化。只有在需要实例时，也就是调用 getInstance 方法时才会去完成对 singleton 实例的实例化。

小总结：

​ 静态内部类的写法与双重检查模式的优点是一样的，都是避免了线程不安全的问题。并且延迟加载、效率高。

​ 静态内部类和双重检查式这两种方法是属于效率比较高的方法，但是它们不能防止被反序列化，生成多个实例。

​ 更高效且安全的方式： 枚举式

4.5 枚举式

执行结果： 执行了单例类的方法，例如返回环境变量信息

​ 借助 JDK 1.5 中添加的枚举类来实现单例模式，这不仅能避免多线程同步的问题，而且还能防止反序列化和反射来创建新的对象来破坏单例的情况的出现。

5. 如何选择最好的单例模式

在前面列出了 5 种实现单例模式的方法，那么我们在实际开发使用哪一种单例最好呢？建议：

​ 最好的单例实现是使用枚举式， Joshua Bloch 在 《Effective java》 一书种明确表达过一个观点.

使用枚举实现单例的方法，虽然还没有被广泛采用，但是单元素的枚举类型已经成为了实现 Singleton 的最佳方法。

​ —— Joshua Bloch

为什么他会更加推崇枚举模式的单例呢?

​ 从枚举写法的优点上看，

首先就是写法简单，枚举的写法不需要我们去考虑懒加载，线程安全等问题。同时代码也比较短小精悍。比其他的写法都更简洁、更优雅。

​ 第二个优点就是线程安全有保障，通过反编译一个枚举类我们可以发现枚举中的各个枚举项是通过 static 代码块来定义和初始化的，它们会在类被加载时完成初始化。

​ 而 Java 类的加载由 JVM 保证线程安全。所以创建一个 Enum 类型的枚举时线程安全的。

​ 而前面的几种单例实现方式存在问题，那就是可能被反序列化破坏，反序列化生成的新的对象从而产生多个实例。

​ 第三个优点是 Enum 能有效防止反序列化（防止破坏单例）

​ Java 专门对枚举的序列化做了规定，在序列化时，仅仅时将枚举对象的 name 属性输出到结果中，在反序列化时，就是通过 java.lang.Enum 的 valueOf 方法来根据名字查找对象，而不是新建一个新的对象。所以，这就防止了反序列化导致的单例破坏问题的出现。

​ 对于通过反射破坏单例而言，枚举类同样由防御措施，反射在通过 newInstance 创建对象时，会检查这个类是否是枚举类，如果时就抛出 IllegalArgumentException(“Cannot reflectively create enum objects” ) 这样的异常，反射创建对象失败。

​ 可以看出，枚举这种方式，可以防止序列化和反射破坏单例。在这一点上与其他的实现方式相比由很多大的优势。安全问题不容小视，一旦生成了多个实例，单例模式就彻底没用了。
