  1. 扩容为2倍，可能考虑到在容量较小时扩容的效率问题

4、String、StringBuffer、StringBuilder 的关系

​ StringBuffer 、StringBuilder 两者的底层与String一样都是字符数组，但是 String 属于不可变字符序列，而StringBuffer、StringBuilder属于可变字符序列；基于可变序列的前提下，StringBuffer是线程安全的，而StringBuilder是线程不安全的。StringBuffer、StringBuilder都是一个容器，最终会通过toString方法变成字符串。

二、为什么有了String，还要有StringBuilder、StringBuffer

​ 在java中字符串是一种引用数据类型，申明的变量实际储存的是一个指向字符串常量值的引用。那么在我们改变用string申明的变量时，实际是重新为一个字符串划分了一个内存，然后改变变量中的引用指向这个新的字符串，而最开始我们指向的旧的字符串是不变的且一直会存在内存中，所以我们常常会对一个字符串进行操作后会新建一个变量（或者对开始的变量重新赋值）来指向他的改变后的值，这样才能确保我们完成了对旧字符串的操作且用变量保存了这个操作的结果。

​ 而对于stringBuffer和StringBuilder申明的变量在内存中则不是这样操作的，它并没有重新划分内存，而是在现在的字符串上进行改变，所以我们不用担心操作后无法得到结果继续其他操作。这样的设计既优化了性能也避免了操作后的结果需要重新申明变量或者重新赋值给原来的变量。

三、String、StringBuilder、StringBuffer的用法

String、StringBuffer

​ String 类型和StringBuffer的主要性能区别：String是不可变的对象, 因此在每次对String 类型进行改变的时候，都会生成一个新的 String 对象，然后将指针指向新的 String 对象，所以经常改变内容的字符串最好不要用 String ，因为每次生成对象都会对系统性能产生影响，特别当内存中无引用对象多了以后， JVM 的 GC 就会开始工作，性能就会降低。

​ 使用 StringBuffer 类时，每次都会对 StringBuffer 对象本身进行操作，而不是生成新的对象并改变对象引用。所以多数情况下推荐使用 StringBuffer ，特别是字符串对象经常改变的情况下。

StringBuilder

​ StringBuilder是5.0新增的。此类提供一个与 StringBuffer 兼容的 API，但不保证同步。该类被设计用作 StringBuffer 的一个简易替换，用在字符串缓冲区被单个线程使用的时候（这种情况很普遍）。如果可能，建议优先采用该类，因为在大多数实现中，它比 StringBuffer 要快。两者的方法基本相同。

使用策略

  • 1) 基本原则：如果要操作少量的数据，用String ；单线程操作大量数据，用StringBuilder ；多线程操作大量数据，用StringBuffer。

四、扩展：”字符串拼接”有几种方式？哪一种效率最高？

开发中，常用的字符串拼接方式有以下几种：

  • 直接使用 + 拼接
  • 使用 String 的 concat() 方法
  • 使用 StringBuilder 的 append() 方法
  • 使用 StringBuffer 的 append() 方法

运行结果如下所示：

1k 1w 10w 100w
+ 11 397 20191 720286
concat 3 72 5671 763612
StringBuilder 0 0 3 17
StringBuffer 1 1 4 36

以上都是执行一次的结果，可能不太严谨，但还是能反映问题的。执行次数越多，性能差距越明显，StringBuilder > StringBuffer > contact > + 。
