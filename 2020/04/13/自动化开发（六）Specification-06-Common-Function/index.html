06.01.01. Equipment Alive

06.01.02. CIM Mode

CIM模式是指定设备与BC之间的通信关系。有两种模式，CIM on 模式和CIM off 模式。设备应在触摸屏上提供CIM模式切换按钮。CIM模式可以改变，无论玻璃计数。

  • CIM on模式:BC和EQ之间通讯正常，EQ可以向BC报告所有事件。BC也可以向EQ发送任何命令
  • CIM OFF模式:BC和EQ之间的通信出现了某种异常，EQ可以更新状态数据( Word 区域)，但不能向BC报告事件( Bit 区域)，不包括设备状态变化报告。同时，装备也不接受BC的任何命令

※注意: 

1. CIM关闭模式，设备应接受BC设置的CIM模式改变命令来改变CIM模式。

CIM Mode ：

​ ON : 机台会把数据上报到BC，BC会收

**OOF** :  机台不会 ON Bit , 只会去写 Word ；BC 只会在 机台 Bit 变化 on 才会去读 Word

06.01.03. CIM Mode Change Command

​ BC可以通过CIM模式改变命令改变CIM模式的设备，进行数据采集。如果设备应答接受，设备需要再次自动触发CIM模式事件

Return Code 说明:

1: Accept，表示设备接受命令，需要改变CIM模式。

2: Already In Desired Status，即CIM模式的指令与设备CIM相同

3: Equipment Reject，意思是设备不能接受指令，不能改变CIM Mode

06.01.05. Inline Mode

内联模式仅与作业转移有关，并且仅在上游设备和下游设备之间必要。设备应按各接口提供触摸屏上的内联模式切换按钮。内联模式包括两种模式:

  • Upstream Inline Mode On (上游内联模式开启) :EQ可以从上游设备接收一个玻璃/芯片。下游内联信号的接口链路信号打开
  • Upstream Inline Mode Off (上游内联模式关闭) :EQ无法从上游设备接收任何玻璃/芯片。下游内联信号的接口链路信号将关闭
  • Downstream Inline Mode On (下游内联模式开启) :EQ可以向下游设备发送一个玻璃/芯片。上游内联信号的接口链路信号将打开。
  • Downstream Inline Mode Off (下游内联模式关闭) :EQ不能向下游设备发送任何玻璃/芯片。上游 Inline 信号的接口链路信号将关闭。

当设备有多个接口时，只要有一组 Inline on，就需要 Inline on向BC报告，当接口全部 Inline off时，就向BC报告 Inline off。

06.03. Equipment Management

06.03.01. Equipment Status

​ 如果设备状态发生变化，必须报告相应的事件。它必须显示在设备的触摸屏上。如果设备状态改变为停止或设置，设备应提前报警，并绘制下报警图。没有直接改变设备状态的远程命令。在BC中，设备状态定义为以下五种类型之一:

  • Idle : 设备不执行进程，但它已经准备好启动进程。这适用于是否有与设备相关的材料。

  • Running : 设备正在执行一个进程。如果工艺完成，没有加工作业，设备又会再次闲置

  • Setup :设备正在初始化模块和实用程序。设备应向BC报告 Alarm Level = Error 。如果设备状态被操作员或BC命令更改为Setup，则报警ID应该报告给BC 65512

  • Stop : 设备进程停止。设备因报警、H/W问题或其他原因停机。在此状态下，设备必须停止工作转移。设备应向BC报告报警级别=报警事件错误。如果设备状态被操作员或BC命令更改为Stop，则报警ID应为report 65513 to BC。

  • Pause : 设备进程暂停。在这种状态下，设备可以立即恢复进程。设备应向BC报告报警级别=报警事件警告。如果设备状态被perator或BC命令更改为Pause，则报警ID应该报告给BC 65514。

    如果设置状态改变，停止和暂停设备必须报告报警ID

※注意:

  1. 当 equipment/unit 状态变为Setup, Stop or Pause 时，设备应将报警事件的报警ID分开(除以BC或EQP异常或EQP正常开关…等)

06.03.02. Unit Status

Unit Status 仅用于多单元设备，最大单元数为15。如果设备没有单元，所有的数据都应该设置为零，将被忽略。单元状态的类型和定义与设备状态的类型和定义相同。每个单元状态是独立的，但是设备状态可能依赖于设备内单元的状态。设备的唯一责任是根据其所在单位的状态报告正确的设备状态

06.04. Alarm Management

alarm (报警）是指设备上可能危及正在加工的人员、设备或物料的任何异常情况。

06.04.01. Alarm Status

报告的是警报状态。

  • Alarm Set： 检测到警报
  • Alarm Clear：警报解除。

06.04.02. Alarm Unit

当设备报告一个警报事件时，它显示该警报与设备中的哪个单元有关。

06.04.03. Alarm ID

所有设备制造商必须提交每台设备的报警标识清单。一个单元必须有一个唯一的警报ID，无论错误或警告警报。

  • 0：reserved (never used)
  • 1 ~ 65500：Equipment can use
  • 65501 ~ 65535：By BC Defined

06.04.04. Alarm Code

  • Bit 00：Danger for human （对人类危险 ）
  • Bit 01：Equipment error
  • Bit 02：Parameter overflow caused process error （参数溢出导致进程错误 ）
  • Bit 03：Parameter overflow caused equipment can’t work （参数溢出导致设备无法工作 ）
  • Bit 04：Cannot recover trouble （无法恢复故障 ）
  • Bit 05：Equipment status warning （设备状态警告 ）
  • Bit 06：Process reached to predefined status （进程达到预定义状态 ）
  • Bit 07：BC Caused

06.04.05. Alarm Level

​ 它显示了警报的级别。它由警告和错误组成。

  • Warning : 警告表示一个灯光错误。对于警告，设备应继续进行处理、蜂鸣操作，并将信息显示在触摸屏上。
  • Error : 告警表示严重错误。当报警发生时，设备必须停止处理(某些设备可能只会导致进程暂停)，蜂鸣操作员对设备进行检查，并在触摸屏上显示信息

06.04.06. Alarm Text

​ 它包含一个简短的文本描述的警报。

06.04.07. Alarm Text Using Flag

​ 它定义了设备或BC使用的报警文本。

06.05. Job Management

​ 这项工作意味着要对玻璃、芯片等材料进行加工。每个作业都有自己独特的数据进行处理，这些数据称为 Job Data，包括PPID、盒式磁带序列号、作业序列号等。 Job Data从上游设备接收，主设备已准备好接收作业，应发送到下游设备，主设备已准备好发送作业。设备应保存和管理每个作业的 Job Data，即使作业在内部区域之间移动。同时，设备应在触摸屏上提供各工种的 Job Data信息。

06.05.01. Job Data Transfer

如果设备向其他设备发送作业，发送作业的设备称为上游设备。如果设备从其他设备接收作业，接收作业的设备称为下游设备。设备可以有一个以上的上游设备和下游设备。为了区分哪些设备是上游设备，哪些是下游设备，对于特定的设备，上游路径号为。被使用和管理。

当向下游设备发送作业时， Job Data通过CC-Link IE随作业一起传输到下游设备。对于 Job Data的传输，设备可以根据设备需要的 Job Data数量，从“发送下游01块 Job Data到”发送下游10块 Job Data，再从“接收上游01块 Job Data”到“接收上游10块 Job Data”。

当上游设备即将向下游设备发送作业时，下游设备首先验证作业的 Job Data。如果在 Job Data中没有检测到错误，下游设备开始接收作业

06.05.05. Store Job Data Event

当设备将作业存储到一个单元或磁带时，应使用端口号和单元编号报告此事件。如果有插槽，则应报告插槽。

06.05.06. Fetch Out Job Data Event

06.05.07. Remove Job Event Mode

​ 当 Job Data 刚刚从设备中删除时，将报告已删除作业的 Job Data。

  • Removed Job : 操作员删除的作业的 Job Data。

  • Unit or Port：它用于区分产品从单元或端口被移除。

  • Unit No： 单元的单元号。如果本机有槽位，设备应更新槽位号

  • Port No：作业被删除的磁带的端口号。设备应更新槽号

  • Slot No：Slot No. 如果 unit 没有 slot ，这个item 应该被清除。

  • Remove Job Flag：

    • Normal Remove： 如果删除作业标志是正常删除，设备应保持删除 Job Data

    • Recovery： 当删除作业标志被删除作业恢复时，设备应恢复此 Job Data。

    • Delete Dirty Data： 如果 Job Data存在，但玻璃丢失，用户可以使用 Job Data删除功能

设备供应商应提供至少10个当前已删除的 Job Data保存功能，并提供 Job Data选择功能进行恢复。

设备供应商应提供至少10个当前已删除的 Job Data的删除保存功能，并提供 Job Data选择功能进行恢复。

如果设备恢复的 Job Data项，如配方号、产品类型、组索引等，这些值的内容与设备本身不同，设备不能改变 Job Data，应发出警告，然后等待操作员处理。

如果BC没有对设备回复信号，对于正常的删除和删除脏数据，意味着在事件复位后， Job Data/计数应该被擦除;对于恢复，这意味着在事件停止后无法恢复 Job Data

06.05.07.01. Error Recovery and Remove Job

当报警或发现作业存在与 Job Data不符，需要更改或删除数据时，设备应具备以下功能:

  • 报警，通知操作员更正

  • 程序”和“作业位置显示”，以恢复上述差异故障或报警情况。

  • 拆卸程序应包含至少三项内容。

  • Physical glass/chip removal

  • Report removed Job Data to BC

  • 从触摸屏设备上删除作业

    设备供应商应提供详细的错误恢复场景。

06.05.07.02 Error Recovery and Normal Process

设备应提供以下功能，用于故障恢复后的正常流程

  • 报警，通知操作员更正
  • Before job transfer signal on : 手动将玻璃送至上游设备
  • After job transfer signal on：通过人工操作将玻璃送至下游设备。
  • Normal process after Error Recovery
  • Report job event to BC

设备供应商应提供详细的错误恢复场景。

06.05.08. Job Count Report

作业计数是报告整个设备的作业计数(不包括卡带)。根据作业类型，设备应区分作业是产品作业还是虚拟作业。 Normally，产品工作意味着大量的产品加工 , 但是在装配线上有两个产品工作， TFT和CF，产品工作意味着TFT玻璃加工 . 设备应根据作业类型的 Job Data实时写入各类作业计数。

  • Total TFT Product Job Count : 它是设备中TFT玻璃的当前计数。
  • Total CF Product Job Count： 这是设备中CF 玻璃的当前计数。
  • Total Dummy Job Count：这是设备中虚拟 玻璃 的数量
  • Total Through Dummy Job Count： 它是在设备中通过虚拟玻璃的计数
  • Total Thickness Dummy Job Count： 它是在设备中的虚拟玻璃厚度的计数
  • Total UV MASK Job Count： 是在设备中UV MASK glasses 的计数
  • special requirement ： 对于一些有单位的设备，应该实时向BC报告作业数。请参阅操作场景规范的每一行。

06.08.03. Equipment Recipe Validation

当 cassette 在端口加载时，在CIM状态下，BC将PPID下载到加载程序，在CIM关闭状态下，操作员在加载程序的触摸屏上输入PPID进行联机操作。

PPID从装载机转移到下游设备，在从上游设备接收玻璃之前，设备应验证PPID(如果启用配方检查)和相应的配方ID。

设备应提供 Job Data Check、 enable/disable 操作选择功能。正常情况下，机器处于 enable 功能;在异常处理情况下，OP选择禁用后，当玻璃从上游传输，下游接收玻璃，不需要检查配方或其他项目，如产品类型，组索引等。

详细规则请参阅CSOT-17_G8.5B_Equipment链路信号通用规范5.4.1 Job Data检查功能。

  1. Auto Recipe Change Enable Case：
  • 如果从上游设备接收的PPID与当前配方ID匹配，则设备可以从上游接收玻璃。
  • 如果从上游设备接收到的PPID与当前的配方ID不匹配，但在配方注册表中找到了，则设备应自动更改配方，从上游设备接收玻璃。
  • 如果从上游设备接收到的PPID与当前配方ID不匹配，且在配方注册表中未找到，则设备应发出警报，且未接收到玻璃。
  • 操作人员在设备上进行配方登记后，设备应再次检查PPID，并自动更改配方。如果从上游设备接收的PPID与当前配方ID匹配，则从上游设备接收玻璃
  1. Auto Recipe Change Disable Case：
  • 如果从上游设备接收的PPID与当前配方ID匹配，则设备可以从上游接收玻璃。
  • 如果从上游设备接收到的PPID与当前配方ID不匹配，则设备应发出警报，不接收玻璃。
  • 在操作员更改当前配方ID后，设备应再次检查PPID。如果从上游设备接收的PPID与当前配方ID匹配，则从上游设备接收玻璃
  1. Auto Recipe Change via Recipe ID Check Mode Case：
  • 配方ID检查模式为禁用(Disable)，配方自动更改为禁用: 无论 Job Data 中的 Recipe ID 与 当前的 设备的Recipe ID 是否相同，都采取从上游设备接收玻璃并使用配方ID的电流进行处理/检测。
  • Recipe ID Check Mode is Enable, Recipe Auto Change is Disable：
    • Job Data 中的 Recipe ID 与 当前设备的 Recipe ID 相同时，从上游设备接收玻璃，使用当前的 Recipe ID 进行加工/检测
    • Job Data 中的 Recipe ID 与 当前设备的 Recipe ID 不同时，不能接收来自上游设备的玻璃并发出警报
  • Recipe ID Check Mode is Disable, Recipe Auto Change is Enable：
    • Job Data 中的 Recipe ID 与 当前设备的 Recipe ID 相同时，从上游设备接收玻璃，使用当前的 Recipe ID 进行加工/检测
    • Job Data 中的 Recipe ID 与 当前设备的 Recipe ID 不同时，应检查 Job Data 中的Recipe ID 是否可以更改:
      • 如果可以更改，则从上游设备接收玻璃，但将配方ID的当前值更改为 Job Data 中的 Recipe ID 进行加工/检测
      • 如果不能更改(包括配方表中不存在配方id)，则无法从上游设备接收玻璃并发出警报。
  • Recipe ID Check Mode is Enable, Recipe Auto Change is Enable：
    • Job Data 上的 Recipe ID 与当前的 Recipe ID 不同时 ，从上游设备接收玻璃，使用当前的 Recipe ID 进行加工/检测。
    • Recipe ID 的 Job Data与 当前Recipe ID 的不同，应检查 的 Job Data 中的Recipe ID 是否可以更改:
      • 如果可以更改，则从上游设备接收玻璃，但将当前的 Recipe ID 更改为Job Data 的Recipe ID 来加工/检测
      • 如果不能更改（包括配方表中不存在 Recipe ID ），则无法从上游设备接收玻璃并发出警报
  • 其他情况:应结合 Job Data Check 项条件(如产品类型、产品ID、组索引启用或禁用)来判断收到的 glass 是否合格。
06.08.03.01. Check Item Description

当上游设备转移 Job 时，设备需要判断是否可以从上游设备接收 Job ，如下:
