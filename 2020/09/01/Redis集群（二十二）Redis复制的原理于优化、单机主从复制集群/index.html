1. 什么是主从复制

1.1 单机有什么问题？

  • 单机如果机器故障，那么就无法及时提供服务；
  • 同时单机的内存等容量有限，无法支持耗能较大的应用；
  • 如果应用的访问并发过大，单机环境无法支持那么大的QPS，就会导致服务宕机等问题。

所以，这时就需要多台服务器主机通过主从复制的机制提供更好的应用。

1.2 主从复制的作用

1） 一主一从模型：

左边的master是主节点，右边的slave是从节点。

​ 如图，假如此时右边是一个空的redis，左边是一个有数据的redis，此时，我做一个复制的操作或复制的命令。那么，左边主节点的数据会向右边传过去。

​ 此时，如果向左边的redis写数据，右边的slave也会有一些操作。

key value key value
K1 V1 K1 V1
K2 V2 K2 V2
K3 V3 K3 V3
…… …… …… ……
Kn Vn Kn

如上表中，左边有K1、K2、K3、…..、Kn，同样的，右边也会有K1、K2、K3、…、Kn。

它能够起到对主机进行备份的作用。可以为主机发生宕机时提供数据的支援。

2） 一主多从模型

​ 一个master可以有多slave，也就是有多个数据的副本。有利于在master和一台slave或两台slave同时挂掉时，还能够有多个数据备份。

​ 以及可以进行读写分离，达到流量分流。

配置项名称 作用 取值

pidfile pid文件位置 /var/run/redis6000.pid

②步骤

  • 第一步：创建/usr/local/cluster-redis目录
  • 第二步：把原始未经修改的redis.conf复制到/usr/local/cluster-redis目录
  • 第三步：把/usr/local/cluster-redis目录下的redis.conf复制为redis6000.conf
  • 第四步：按照既定计划修改redis6000.conf中的相关配置项
    • dir
    • port
    • dbfilename
    • logfile
    • pidfile
  • 第五步：复制redis6000.conf为redis7000.conf
  • 第六步：修改redis7000.conf中的相关配置项
    • port
    • dbfilename
    • logfile
    • pidfile
  • 第七步：复制redis6000.conf为redis8000.conf
  • 第八步：修改redis8000.conf中的相关配置项
    • port
    • dbfilename
    • logfile
    • pidfile

③启动Redis主从复制集群

启动对应端口号的客户端： /usr/local/redis/bin/redis-cli -p 6000 (这里要指定端口号，用于访问对应的客户端)

使用redis-cli停止指定服务器的命令格式如下：

/usr/local/bin/redis-cli -h IP地址 -p 端口号 shutdown
