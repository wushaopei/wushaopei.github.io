api - bundle

基本原则：

  1. Port的获取

一条线上可能有多个设备，每个设备上有unit，每个unit或设备上有job。如果要找PORT，就直接找PortDAO。

Port通过属性和EQP关联起来，****

  1. 捞取数据

有一部分数据是从DB捞的，有一部分是从实时运行中捞的数据；从DB上捞的数据也就是我们做的BC工具里面的数据，我们都是放在 Data 中的。

注意：

凡是以BC * 开头的参数都是可以从Data中获取数据的，例如： BcPort、BcEquiment、BcUnit等。

这些数据都是在DB里面捞的，这些数据都是在我们 set Off 之前，key 的数据库里面的数据。

应用：

如果要查找Port的一些属性，比如： Port属于哪个设备、哪条线，都可以通过 getData 去取里面的属性就可以了。

数据实体对应关系

DB对应实体：

BcPort、BcEquiment 等Bc开头的实体都对应数据库的表，主要放在 com.glorysoft.bc.db.entity包

逻辑层对应实体：
放置在com.glorysoft.bc.entity包下

在EquipmentDaoImpl 类中，存在一个 Map<String,Equipment>的集合类。如果一条线中存在有四个设备，那么这个 equipmentmap 中就有四个Equipment 。

bc-service- * - bundle

bc-service-machine-bundle

对OSGI的实现

业务层中每个bundle必须实现一个OSGI提供的接口 - BundleActivator ,并实现该类的方法。

重写的方法为 start() 和 stop() 这两个方法，其中的 getContext() 和 getExecutor () 两个方法是用于获取上下文和配置所用的。

start(BundleContext context)

在包启动过程中，会启动 bundleContext = context，同时获取上下文的数据。

服务注册

在 bc-service- * -bundle 中声明了服务之后，如 AlarmServiceImpl等之后如果不进行注册，就无法被外界访问到。这一步很重要，这里通过 registerService（） 方法将服务注册到OSGI，从而实现了接口的对外开放。

服务监听

这里主要是对服务进行监听，其监听机制是动态添加服务，即 addService（）进行添加服务，每次启动都会将服务添加到 ServiceHolder中去。

服务的调用

在其他的逻辑层代码中，只需要通过 ServiceHolder就可以调用到对应的服务了。

stop(BundleContext context)

在关闭的时候会调用类中的 alarmService.unregister(); 方法。

以上两者可以理解为，启动时对资源进行获取和占用，关闭时需要对资源进行释放。

bc-service-recipe-bundle

涉及的业务逻辑有：

  1. 有三层timeout;
  2. 有三层recipe检查、MES检查、BC检查
  3. 以及多个方向： 如MES到BC，BC到设备，最后设备回回来，设备回回来还要和MES请求，mes再回给设备。
